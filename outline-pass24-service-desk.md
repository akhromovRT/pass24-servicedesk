## Slide 1: Заголовок
- PASS24 Service Desk
- Единая точка входа для технической поддержки клиентов PASS24
- Презентация проекта для команды
- Дата: апрель 2026

## Slide 2: Контекст — что такое PASS24
- PASS24.online — облачная СКУД-система для жилых комплексов и бизнес-центров
- Управление пропусками, доступом к территории, шлагбаумами, домофонами
- Тысячи пользователей: жители, управляющие компании, администраторы объектов
- Все эти люди периодически нуждаются в технической поддержке

## Slide 3: Проблема — почему нужен Service Desk
- Нет единой точки входа для обращений клиентов
- Заявки теряются в чатах, звонках и письмах
- Нет прозрачности: клиент не знает статус своего обращения
- Нет базы знаний — одни и те же вопросы решаются заново
- Поддержка не может приоритизировать и отслеживать SLA

## Slide 4: Решение — PASS24 Service Desk
- Help Desk портал для клиентов и партнёров PASS24
- Два ключевых модуля: система тикетов + база знаний
- Самообслуживание: клиенты сами находят ответы в FAQ и инструкциях
- Прозрачность: полный жизненный цикл заявки виден всем участникам

## Slide 5: Пользователи системы
- **Жители / резиденты ЖК и БЦ** — создают заявки по проблемам с пропусками, доступом, приложением
- **УК / администраторы объектов** — управляют заявками от своих резидентов, ищут решения в базе знаний
- **Команда поддержки PASS24** — обрабатывает тикеты, наполняет базу знаний
- **Системные администраторы** — настраивают роли, категории, правила маршрутизации

## Slide 6: Архитектура — текущее состояние и целевая архитектура
- **Сейчас (прототип):**
  - Монолитное приложение: серверный рендеринг шаблонов + встроенные скрипты
  - Хранение данных: in-memory (словарь в памяти процесса), без внешних зависимостей
  - Кликабельный UI-макет (тёмная тема)
  - Запуск одной командой — работает из коробки
- **Целевая архитектура (MVP):**
  - Клиент-серверная архитектура: SPA + REST API (разделение frontend и backend)
  - Компонентный UI вместо монолитного шаблона
  - Реляционная СУБД вместо in-memory хранилища
  - RBAC: 4 роли (resident, admin УК, support agent, system admin)

## Slide 7: Доменная модель — тикет
- Основная сущность: Ticket (id, title, description, category, status, priority)
- Категории: general, access, pass, gate, notifications, other
- Приоритеты: LOW, NORMAL, HIGH, CRITICAL
- Связанные поля: object_id (ЖК/БЦ), access_point_id (дверь/шлагбаум), user_role, urgent
- События (TicketEvent) — полный аудит-трейл: кто, когда, что изменил

## Slide 8: Автоприоритет — пример бизнес-логики
- Автоматический анализ ключевых слов в заголовке и описании заявки
- **CRITICAL** — "не могу попасть", "не пускает", "дверь не открылась"
- **HIGH** — "шлагбаум", "парковка"
- **LOW** — "уведомлен", "пуш"
- **NORMAL** — всё остальное (приоритет по умолчанию)
- Снижает нагрузку на оператора, ускоряет реакцию на критичные случаи
- Покрыто юнит-тестами — легко расширять словарь ключевых слов

## Slide 9: State Machine — граф переходов статусов
- Граф допустимых переходов (не линейная цепочка):
  - **NEW** → IN_PROGRESS, RESOLVED
  - **IN_PROGRESS** → WAITING_FOR_USER, RESOLVED
  - **WAITING_FOR_USER** → IN_PROGRESS, RESOLVED
  - **RESOLVED** → CLOSED, IN_PROGRESS (переоткрытие)
  - **CLOSED** — терминальное состояние, переходы запрещены
- Валидация на уровне доменной модели — невозможно перескочить или вернуться в некорректный статус
- Каждый переход фиксируется как TicketEvent с actor_id и timestamp

## Slide 10: Технический стек
- **Backend:**
  - **Python 3.12** — актуальная версия с улучшенной типизацией
  - **FastAPI** — async из коробки, автогенерация Swagger/OpenAPI документации
  - **SQLModel** — единая модель для Pydantic-валидации и SQLAlchemy ORM
  - **PostgreSQL 16** — надёжная реляционная БД с полнотекстовым поиском
  - **pytest** — покрытие тестами с первого дня
- **Frontend:**
  - **Vue 3** — Composition API, реактивность, отличная экосистема
  - **TypeScript** — строгая типизация, меньше ошибок на этапе разработки
  - **PrimeVue** — enterprise-ready UI-компоненты (таблицы, формы, диалоги)
  - **Pinia** — управление состоянием приложения
- Обоснование выбора зафиксировано в ADR-001

## Slide 11: Демо — UI-прототип
- Live demo: запуск `uvicorn backend.main:app` и показ интерфейса
- Кликабельный макет: категории заявок, форма создания тикета, личный кабинет, FAQ
- Тёмная тема, адаптивная вёрстка
- Прототип демонстрирует целевой UX до разработки Vue 3 SPA

## Slide 12: Текущий статус — что уже сделано
- Backend API skeleton на FastAPI — рабочий прототип
- Доменные модели: Ticket, TicketEvent, Category, Status, Priority
- REST endpoints тикетов (создание, список, просмотр, смена статуса)
- Автоприоритет по ключевым словам
- State machine с валидацией переходов
- Unit тесты: 6 тестов на pytest, все проходят
- UI mockup: Jinja2 + vanilla JS, тёмная тема
- In-memory storage — работает без внешних зависимостей

## Slide 13: Roadmap — план развития
- **v0.1 MVP** — PostgreSQL + SQLModel ORM, Vue 3 SPA (замена Jinja2), RBAC, email-уведомления
- **v0.2** — полнотекстовый поиск по базе знаний, аналитика (vue-echarts), дашборд для УК
- **v0.3** — SLA tracking, файловые вложения к тикетам, внутренние комментарии для команды поддержки
- Итеративный подход: каждая версия — рабочий инкремент
