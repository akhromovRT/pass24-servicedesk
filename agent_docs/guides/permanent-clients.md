# Постоянные клиенты в Service Desk — руководство для агента

Описание: что такое «постоянный клиент», как он попадает в Service Desk из Bitrix24 и как пользоваться этим в повседневной работе с заявками.

Целевая аудитория: агенты поддержки, администраторы.

---

## Кратко

- В Bitrix24 у компании есть чекбокс **«Постоянный клиент»** (`UF_CRM_PERMANENT_CLIENT`). Если он стоит — клиент считается постоянным.
- Service Desk **каждую ночь в 03:00** забирает таких клиентов и кладёт в свой справочник `customers` (ключ синхронизации — ИНН).
- Заявка может быть привязана к компании через `customer_id`. Если эта компания помечена как постоянная — на заявке появляется **золотой бейдж «Постоянный»**.
- В списке заявок есть тогл **«Постоянные клиенты / Все клиенты»** — отфильтровать ленту до заявок только от постоянных клиентов.

---

## 1. Откуда берётся признак

Источник правды — Bitrix24, портал `pass24pro.bitrix24.ru`.

1. Менеджер Bitrix24 ставит чекбокс «Постоянный клиент» на карточке компании.
2. Раз в сутки (03:00) Service Desk дёргает `crm.company.list`, забирает все компании с этим флагом и сопоставляет их с локальной таблицей `customers` по ИНН (получает его из `crm.requisite.list`, поле `RQ_INN`).
3. Если компании нет в `customers` — создаётся; если есть — обновляется (имя, адрес, телефон, email, отрасль, флаг «постоянный»).

**Важно:** компании без ИНН в Bitrix24 при синхронизации пропускаются (≈2 компании из 232 на 2026-04). Это сознательное ограничение — ИНН выбран как единственный устойчивый ключ синхронизации (см. ADR-008).

Один клиент может быть «постоянным» сегодня и перестать им быть завтра — флаг просто перепишется при следующей синхронизации.

### Если sync ещё не прошёл

После того как менеджер поставил чекбокс в Bitrix24, изменение появится в Service Desk **на следующий день после 03:00**. Если нужно срочно — администратор Service Desk может запустить синхронизацию вручную:

```
POST /customers/sync          # admin only, в фоне
```

или через UI (в админке, раздел синхронизации).

## 2. Где это видно в интерфейсе

Бейдж «Постоянный» (золотая звёздочка на жёлтом фоне) появляется в трёх местах:

### 2.1. Автокомплит при создании заявки

Когда агент создаёт заявку и вводит название компании в поле **«Компания-клиент»**, в выпадающем списке постоянные клиенты:
- **Идут в начале списка**, до обычных компаний.
- **Помечены бейджем «Постоянный»** прямо в строке.

Под автокомплитом, если в локальном справочнике мало совпадений (меньше 3 при запросе ≥ 3 символов), подгружаются результаты из реестра ФНС (DaData) — это уже не «постоянные», добавляются по клику.

### 2.2. Карточка заявки (правая панель «Объект»)

В блоке **«Клиент / Объект»**:
- Иконка `🔗` (зелёная) — заявка привязана к компании из справочника.
- Бейдж **«Постоянный»** — компания помечена как постоянный клиент.

Эти два индикатора независимы: можно быть привязанным к компании из справочника, но не быть постоянным клиентом.

### 2.3. Список заявок (мета-строка карточки)

В строке тикета, в meta-области рядом с автором и каналом, для постоянных клиентов отображается:
```
… · ⭐ Постоянный · …
```

## 3. Фильтр «Только постоянные клиенты»

В шапке списка заявок (только для агентов и админов) есть тогл-кнопка:
- **«Все клиенты»** (серый, по умолчанию) — заявки от любых клиентов.
- **«Постоянные клиенты»** (жёлтый) — только заявки, где `customer_id` ведёт на компанию с `is_permanent_client = true`.

Нажатие переключает режим и сразу перезагружает список. Кнопка **«Сбросить»** рядом с фильтрами по статусу/категории сбрасывает в том числе этот тогл.

Когда использовать:
- Утренний обход заявок от ключевых клиентов.
- Подготовка отчётности — «сколько обращений от постоянных клиентов за неделю».
- Эскалация — постоянным клиентам обычно нужен повышенный приоритет и более быстрая реакция.

## 4. Как привязать заявку к постоянному клиенту

### При создании заявки

1. Выбрать тему через quick-card → форма Step 2.
2. В блоке **«Компания-клиент»** начать вводить название.
3. Выбрать постоянного клиента (с бейджем) из выпадающего списка.
4. Если в списке нет — нажать **«+»**, ввести ИНН → данные подтянутся из ФНС → создастся обычная (не постоянная) запись в `customers`.

После создания `customer_id` заявки сохраняется в БД.

### В существующей заявке

1. Открыть карточку, в правой панели найти блок **«Объект»**.
2. Нажать **«Изменить»** или **«Указать клиента»**.
3. В автокомплите начать вводить название — **видны только постоянные клиенты** (фильтр `permanent_only=true`).
4. Выбрать клиента, при необходимости отредактировать адрес и точку доступа.
5. Нажать **«Сохранить»** — в журнал событий заявки добавится запись об изменении.

> Различие в логике поиска намеренное: при создании заявки агент часто сталкивается с разовым клиентом (не постоянный, нужно срочно зарегистрировать через ИНН). При редактировании уже существующей заявки логичнее показывать только постоянных — это обычно правка ошибки или эскалация.

## 5. API (для интеграций)

| Эндпоинт | Что возвращает |
|---|---|
| `GET /customers/search?q=...` | Список клиентов; каждый элемент содержит `is_permanent_client: boolean`. Постоянные сортируются в начало. |
| `GET /customers/search?q=...&permanent_only=true` | То же, но только постоянные. |
| `GET /tickets/objects/suggest?q=...` | Только постоянные клиенты, для автокомплита в форме редактирования объекта. |
| `GET /tickets/?customer_only_permanent=true` | Список заявок, привязанных к постоянным клиентам. Дополнительно поддерживается `customer_id=<uuid>` — заявки конкретной компании. |
| `GET /tickets/{id}` | В `TicketRead` поле `customer_is_permanent: boolean \| null`. `null` — если у заявки нет `customer_id` или Customer не найден. |
| `POST /customers/sync` (admin) | Запускает синхронизацию с Bitrix24 в фоне. |

Контракт `Customer`: id, inn, name, bitrix24_company_id, address, phone, email, industry, is_active, **is_permanent_client**, synced_at, created_at.

## 6. Частые вопросы

**Менеджер поставил чекбокс в Bitrix24 — в Service Desk обновится через сколько?**
Через 24 часа максимум (на следующее утреннее окно 03:00). Срочно — попросить администратора Service Desk дёрнуть `POST /customers/sync`.

**Клиент с тем же названием уже есть в Service Desk, но без флага.**
Sync найдёт его по ИНН и просто проставит флаг — дубль не создастся. Если ИНН отличается — появится вторая запись. ИНН у юрлиц устойчив, у ИП — может меняться (редко).

**Заявка привязана к компании, но бейджа нет.**
Значит компания есть в `customers`, но `is_permanent_client = false`. Проверьте Bitrix24: стоит ли там чекбокс. Если стоит — дождитесь следующего sync или попросите ручной запуск.

**Хочу видеть конкретного клиента, а не всех постоянных.**
Используйте параметр `customer_id` в API. В UI пока нет отдельного селектора компании в фильтрах списка — это в roadmap.

**Резидент не видит бейджа.**
Бейдж в meta-строке карточки и тогл «Только постоянные» доступны только staff-ролям (`support_agent`, `admin`). Резиденту в его собственной заявке бейдж в правой панели «Объект» виден, если заявка к постоянному клиенту привязана и резидент имеет к ней доступ.

**Поле `UF_CRM_PERMANENT_CLIENT` называется по-другому в Bitrix24.**
В коде есть резерв — функция `discover_inn_field` ищет поле по подстроке «инн/inn» в title. Для permanent-клиентов аналогичный fallback пока не реализован: имя поля захардкожено в `crm.company.list` select. Если на портале поле переименовали — нужно поправить `backend/customers/bitrix24_sync.py:143`. Заведите задачу разработке.
