# База знаний PASS24 Service Desk — руководство для менеджеров

> Документ описывает, как устроена и работает база знаний в портале технической поддержки support.pass24pro.ru. Для менеджеров технической поддержки, руководителей и owner'ов портала.

---

## Зачем нужна база знаний

**База знаний (БЗ) — это витрина готовых ответов** на частые вопросы клиентов. Её задача — **снять нагрузку с менеджеров ТП**, дать клиентам возможность найти решение самостоятельно за 1-2 минуты.

По данным стратегического отчёта ТП за 2025 год, **до 60-70% типичных обращений можно автоматизировать** через грамотную базу знаний. Самые частые темы:

| Тема | Заявок/год | % потока | Можно автоматизировать |
|---|---|---|---|
| SMS-коды для регистрации | 247 | 7.1% | 85% |
| SMS-коды для восстановления | 124 | 3.6% | 85% |
| Пропуска и шлагбаумы | 59 | 1.7% | 55% |
| Распознавание номеров | 21 | 0.6% | 30% |

**Каждая статья в БЗ, которая реально помогает клиенту — это минус одна заявка в ТП.**

## Как клиент видит базу знаний

### Главная страница (support.pass24pro.ru/knowledge)

Клиент попадает на базу знаний **тремя путями:**

1. **Прямой визит** — через меню «База знаний» на портале
2. **При создании заявки** — система автоматически показывает релевантные статьи при вводе заголовка
3. **Из email-ответа** — если клиент написал на support@pass24online.ru, в автоответе приходят 3 релевантные статьи

На главной странице он видит:

- **Большое поле поиска** — главный инструмент. Клиент пишет вопрос своими словами: «не приходит смс», «забыл пароль», «прога крашится» — и система находит нужные статьи.
- **Быстрый фильтр** — 3 кнопки: «Все / Инструкции / FAQ»
- **Топ-теги** — кликабельные ярлыки с количеством статей (например, «SMS-коды (2)», «Регистрация (3)»). Клик → мгновенно фильтрует.
- **Секции по категориям** — статьи сгруппированы по продуктам PASS24:
  - Мобильное приложение
  - Пропуска и QR-коды
  - Доступ и вход
  - Шлагбаумы и камеры
  - Уведомления
  - Общее

### Страница статьи

Открыв статью, клиент видит:

- **Заголовок + теги** — сразу понятно, о чём статья
- **Дата обновления + счётчик просмотров** — видно, что статья живая
- **Содержание (TOC)** — для длинных статей автоматически строится оглавление из подзаголовков
- **Основной контент** — пошаговая инструкция с выделенными блоками «Важно» (жёлтый) и «Совет» (зелёный)
- **Кнопка «Помогла ли статья?»** — 👍 или 👎 с опциональным комментарием
- **Связанные статьи** — до 4 статей из той же категории
- **CTA «Не помогло — создать заявку»** — если решение не нашлось

## Как это помогает клиенту: 3 сценария

### Сценарий 1: Клиент пишет в форму заявки
1. Начинает вводить заголовок: «не приходит смс при регистрации»
2. Справа всплывают 3 подходящие статьи из БЗ
3. Клиент кликает → читает → находит решение
4. **Заявка не создаётся**. Менеджеру не нужно отвечать — проблема решена.

### Сценарий 2: Клиент пишет на support@pass24online.ru
1. Присылает письмо: «У меня не приходит SMS, как мне войти?»
2. Система автоматически (в течение 60 секунд) ищет в БЗ
3. Отправляет клиенту ответ: «Возможно, эти статьи помогут» + 3 ссылки
4. Параллельно создаётся тикет — если статьи не помогут, менеджер его подхватит

### Сценарий 3: Клиент зашёл на сайт
1. Открывает /knowledge, пишет «забыл пароль»
2. Поиск выдаёт топ-3 статьи (даже без точных слов благодаря синонимам)
3. Клиент читает, нажимает 👍 «Помогла»
4. **Метрика deflection** растёт — руководство видит пользу БЗ в цифрах.

## Для агентов технической поддержки

### Что видит агент на странице тикета

На странице заявки (для staff) есть **панель «Связанные статьи»**:
- Поиск по БЗ прямо из тикета
- Привязка найденной статьи к тикету с указанием типа связи:
  - `helped` — «эта статья помогла решить»
  - `related` — «по теме»
  - `created_from` — «клиент пришёл из этой статьи, но не нашёл ответа»

Когда клиент нажимает **«Не помогло — создать заявку»** на странице статьи, в тикете **автоматически** появляется связь `created_from` и в истории заявки запись: «Клиент пришёл из статьи «...» (не нашёл ответа)».

### Создание и редактирование статей

Агент с ролью `support_agent` или `admin` может:
1. Создать статью через кнопку «Создать статью» на главной /knowledge
2. Указать: заголовок, категорию, тип (FAQ/Инструкция), содержимое в Markdown
3. Добавить **теги** (для поиска по темам)
4. Добавить **синонимы** — альтернативные формулировки клиентов

**Синонимы — это ключевая фишка**. Клиенты пишут по-разному: «смс не приходит», «нет кода», «не получаю SMS». Если в поле synonyms статьи указать все эти варианты — поиск найдёт статью независимо от того, как клиент сформулировал.

## Для руководителя ТП: аналитика базы знаний

На странице **/kb-analytics** (доступна для агентов и админов, кнопка «Аналитика» на главной БЗ) показываются метрики.

### 4 ключевые карточки

1. **Просмотры статей** — сколько раз клиенты открывали статьи БЗ
2. **Helpful ratio** — % клиентов, нажавших 👍 (целевое значение ≥ 60%)
3. **Deflection Rate** — % клиентов, решивших вопрос без создания заявки (главная KPI метрика)
4. **Заявок из статей** — сколько клиентов прочитали статью и **всё равно создали заявку** (сигнал: контент нужно дорабатывать)

### Таблица «Топ помогающих статей»

Статьи с наибольшей долей 👍 (минимум 3 отзыва). Это ваши **лучшие активы** — их надо показывать первыми, рекомендовать в обучении, использовать как шаблоны для новых статей.

### Таблица «Статьи требующие внимания»

Статьи, после чтения которых клиенты **всё равно создали заявку**. Это сигналы:
- Статья неполная — не хватает какого-то шага
- Статья не по теме — клиент пришёл с одной проблемой, а статья про другую
- Формулировка вводит в заблуждение

**Что с ними делать:** читать эти статьи + смотреть связанные тикеты → переписывать контент. Повторная проверка через 2 недели.

## Структура базы знаний: категории и типы

### 2 типа статей
- **FAQ** (часто задаваемые вопросы) — короткие ответы на конкретные проблемы
- **Инструкция (Guide)** — пошаговое руководство (установка, настройка)

### 6 категорий
| Категория | Назначение |
|---|---|
| **Доступ и вход** (access) | Регистрация, SMS, пароли, авторизация |
| **Пропуска и QR** (pass) | Создание пропусков, гостевые QR-коды |
| **Шлагбаумы и камеры** (gate) | Распознавание номеров, проезд через КПП |
| **Мобильное приложение** (app) | iOS/Android, установка, сбои |
| **Уведомления** (notifications) | Push, email, настройки |
| **Общее** (general) | О платформе, безопасность, администрирование |

### Теги
Второй уровень классификации — **теги**. Одна статья может иметь несколько тегов. Примеры:
- `sms` + `registration` + `onboarding` — первая регистрация клиента
- `sms` + `recovery` + `password` — восстановление пароля
- `login` + `troubleshooting` + `hub` — общий гайд по входу

Теги помогают клиенту быстро фильтровать статьи по теме.

## Как писать хорошую статью для базы знаний

### 1. Заголовок — это главное

Заголовок должен **точно отражать проблему клиента**, его словами. Не:
- ❌ «Аутентификация через SMS-OTP»

А:
- ✅ «Не приходит SMS-код при регистрации»

**Почему:** клиент ищет по тому, что происходит **у него**, а не по техническим терминам.

### 2. Структура статьи: проблема → шаги → эскалация

Каждая статья должна отвечать на 3 вопроса:
1. **Что происходит?** (симптомы, когда это случается)
2. **Что делать по порядку?** (3-5 пошаговых действий)
3. **Что если не помогло?** (как создать заявку, какую информацию указать)

### 3. Используйте выделения

Markdown-синтаксис поддерживает специальные блоки:
- `> **Важно:** текст` → жёлтый блок-предупреждение
- `> **Совет:** текст` → зелёный блок-подсказка
- `> **Внимание:** текст` → красный блок с предупреждением

Эти блоки автоматически визуально выделяются. **Но не злоупотребляйте** — 1-2 на статью, иначе теряется акцент.

### 4. Короче и по делу

- **Длина FAQ-статьи:** 300-800 слов
- **Длина инструкции:** 500-1500 слов
- **Шаги:** нумерованный список, одно действие = один шаг
- **Подзаголовки:** 3-5 H2 на статью

### 5. Обязательно: теги и синонимы

- **Теги:** 2-4 штуки, из существующего набора (посмотрите топ-теги на главной БЗ)
- **Синонимы:** 3-6 фраз, как клиенты реально говорят о проблеме
  - Пример для статьи «Неверный пароль»: `["не могу войти", "пароль не подходит", "не входит", "ошибка пароля"]`

## Целевые метрики на 2-3 месяца (май-июль 2026)

| Метрика | Базовое значение | Цель |
|---|---|---|
| **Deflection Rate** (все каналы) | ~0% (не измерялся) | 15-25% |
| **Helpful Ratio топ-10 статей** | нет данных | ≥ 60% |
| **Заявки по SMS-теме** | 371/год | 230/год (-38%) |
| **Время обработки типичной заявки** | 20-30 мин | 5-10 мин |
| **Нагрузка на менеджеров** | 100% | 30-40% |

**ROI:** внедрение занимает 2-3 месяца → экономит **~50-70%** рутинной работы менеджера.

## Частые вопросы менеджеров

**В: Кто должен писать статьи?**  
О: Любой агент ТП. Лучше всего пишет тот, кто только что закрыл 3-5 похожих тикетов — он помнит формулировки клиентов.

**В: Как узнать, что нужна новая статья?**  
О: На дашборде /kb-analytics в таблице «Статьи требующие внимания» смотрите статьи с высоким показателем «заявок из статей». Если таких много — контент нужно переписать или разбить на несколько статей.

**В: Как узнать о новых темах, для которых нет статей?**  
О: Анализируйте тикеты с категорией «Другое» (82% потока сидит там). Если вы закрываете одну и ту же проблему больше 3 раз — это кандидат на новую статью.

**В: Можно ли удалять старые статьи?**  
О: Нет. Лучше **обновить** статью (через кнопку «Редактировать»). Система автоматически сохраняет старый URL в aliases — ссылки из email и закладок клиентов продолжают работать.

**В: Как клиент находит статью, если не знает точного названия?**  
О: Благодаря **синонимам** и **Russian stemmer** в PostgreSQL FTS. Клиент пишет «прога крашится» — находит статью «Приложение PASS24 вылетает» (через synonyms `"прога"→"приложение"`, `"крашится"→"вылетает"`).

**В: Можно ли сделать статью только для внутреннего пользования?**  
О: Да, снимите галочку «Опубликовано». Статья останется в системе, но клиентам не будет видна. Используется для draft'ов и внутренних инструкций.

## Контрольный список менеджера при создании статьи

- [ ] Заголовок написан словами клиента (не техническими)
- [ ] Указана правильная категория
- [ ] Выбран тип: FAQ (ответ) или Guide (инструкция)
- [ ] Добавлено 2-4 тега
- [ ] Добавлено 3-6 синонимов (как клиенты реально говорят)
- [ ] Структура: Проблема → Шаги → Эскалация
- [ ] 1-2 блока `> **Важно:**` или `> **Совет:**` с ключевой информацией
- [ ] В конце — инструкция что делать, если не помогло (указать контакты)
- [ ] Через 2 недели после публикации — проверить helpful_ratio на дашборде
