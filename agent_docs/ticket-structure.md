# Оптимальная структура заявки PASS24 Service Desk

> На основе анализа 3 454 реальных заявок за 2025 год, SLA-регламента и классификации обращений.

---

## Проблема текущей структуры

Текущая модель имеет одно поле `category` (6 значений), в результате чего **82% заявок (2 833 шт.)** попадают в категорию «Другое». Это делает невозможными:
- Автоматическую маршрутизацию
- Аналитику по типам проблем
- Корректный SLA
- Автоматизацию типовых ответов

---

## Предлагаемая структура

### 1. Классификаторы (5 осей)

#### 1.1. Продукт (`product`)
*Что затронуто — какой продукт PASS24*

| Значение | Название | % заявок |
|----------|----------|----------|
| `pass24_online` | PASS24.online (приложение и веб-портал) | 89.8% |
| `pass24_key` | PASS24.Key (BLE мобильные ключи) | 0.7% |
| `pass24_control` | PASS24.control (СКУД-контроллер) | 0.4% |
| `pass24_auto` | PASS24.auto (распознавание номеров) | 0.6% |
| `equipment` | Оборудование (считыватели, камеры, контроллеры) | 2.1% |
| `integration` | Интеграция (Sigur, TRASSIR, ZKTeco, ivideon) | 1.1% |
| `other` | Другое / Общие вопросы | 5.3% |

#### 1.2. Категория проблемы (`category`)
*Что именно сломалось — уточняет продукт*

| Значение | Название | Заявок | % |
|----------|----------|--------|---|
| `registration` | Регистрация и вход | 412 | 11.9% |
| `passes` | Пропуска и доступ | 59 | 1.7% |
| `recognition` | Распознавание номеров | 21 | 0.6% |
| `app_issues` | Работа приложения (вылеты, ошибки) | 30 | 0.9% |
| `objects` | Добавление/настройка объектов | 37 | 1.1% |
| `trusted_persons` | Доверенные лица | 13 | 0.4% |
| `equipment_issues` | Неисправность оборудования | 19 | 0.6% |
| `consultation` | Консультация по продукту | 30 | 0.9% |
| `feature_request` | Предложение по улучшению | 9 | 0.3% |
| `other` | Другое | — | — |

#### 1.3. Тип обращения (`ticket_type`)
*Зачем пишут — влияет на SLA и маршрутизацию*

| Значение | Название | SLA по умолчанию |
|----------|----------|-----------------|
| `incident` | Инцидент (массовый сбой, всё не работает) | 4 раб. часа |
| `problem` | Проблема (не работает у меня) | 8 раб. часов |
| `question` | Вопрос / Консультация | 24 раб. часа |
| `request` | Запрос на настройку / изменение | 24 раб. часа |
| `feature_request` | Предложение / Идея | 5 раб. дней |

#### 1.4. Приоритет (`priority`)
*Насколько срочно — рассчитывается автоматически, может быть изменён оператором*

| Значение | Название | SLA реакция | SLA решение | Когда назначается |
|----------|----------|------------|-------------|-------------------|
| `critical` | Критический | 30 мин | 4 раб. часа | Полная остановка системы, невозможно попасть на объект |
| `high` | Высокий | 1 час | 8 раб. часов | Шлагбаум не открывается, проблемы с доступом |
| `normal` | Обычный | 4 часа | 24 раб. часа | Ошибки приложения, проблемы с пропусками |
| `low` | Низкий | 8 часов | 48 раб. часов | Консультации, уведомления, предложения |

**Автоприоритет** (по ключевым словам + ticket_type):
- `incident` → `critical`
- "не могу попасть", "не пускает", "заблокирован" → `critical`
- "шлагбаум", "ворота", "не работает" → `high`
- `question`, `feature_request` → `low`
- Всё остальное → `normal`

#### 1.5. Источник (`source`)
*Откуда поступила заявка — заполняется автоматически*

| Значение | Название | % заявок |
|----------|----------|----------|
| `web` | Веб-портал Service Desk | — (новый канал) |
| `email` | Email (support@pass24online.ru) | 59.2% |
| `telegram` | Telegram-бот | 29.9% |
| `api` | API / Admin24-SaaS | 10.9% |
| `phone` | Телефон | — |

### 2. Поля заявки

#### 2.1. Основные (заполняет заявитель)

| Поле | Тип | Обязательное | Описание |
|------|-----|:---:|----------|
| `title` | string(200) | Да | Тема заявки |
| `description` | text(4000) | Да | Подробное описание проблемы |
| `product` | enum | Да | Какой продукт затронут |
| `category` | enum | Нет* | Категория проблемы (*авто-определяется) |
| `ticket_type` | enum | Нет* | Тип обращения (*по умолчанию `problem`) |
| `urgent` | bool | Нет | «Не могу попасть / проехать прямо сейчас» |

#### 2.2. Контактные данные

| Поле | Тип | Обязательное | Описание |
|------|-----|:---:|----------|
| `contact_name` | string(256) | Да | ФИО заявителя |
| `contact_email` | email | Да | Email для уведомлений |
| `contact_phone` | string(20) | Нет | Телефон для связи |
| `company` | string(256) | Нет | Компания / УК |

#### 2.3. Объект (где проблема)

| Поле | Тип | Обязательное | Описание |
|------|-----|:---:|----------|
| `object_name` | string(256) | Рекомендуемое | Название объекта (ЖК Солнечный, БЦ Меридиан) |
| `object_address` | string(512) | Нет | Адрес (улица, дом, корпус) |
| `access_point` | string(128) | Нет | Точка доступа (КПП-1, подъезд 3, парковка) |

#### 2.4. Техническая информация (необязательные, помогают диагностике)

| Поле | Тип | Описание |
|------|-----|----------|
| `device_type` | enum: `ios`, `android`, `web`, `other` | Тип устройства |
| `app_version` | string(32) | Версия приложения PASS24 |
| `os_version` | string(32) | Версия iOS/Android |
| `error_message` | string(500) | Текст ошибки (если есть) |
| `screenshot` | file[] | Скриншоты проблемы |

#### 2.5. Служебные (заполняются системой / оператором)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | uuid | Уникальный ID |
| `creator_id` | FK → User | Кто создал |
| `assignee_id` | FK → User, nullable | Ответственный менеджер |
| `status` | enum | Статус (FSM) |
| `priority` | enum | Приоритет (авто + ручной override) |
| `source` | enum | Откуда пришла заявка |
| `sla_response_hours` | int | SLA на первый ответ (из приоритета) |
| `sla_resolve_hours` | int | SLA на решение (из приоритета) |
| `first_response_at` | datetime | Когда первый раз ответили |
| `resolved_at` | datetime | Когда решили |
| `sla_breached` | bool | Нарушен ли SLA |
| `sla_paused_at` | datetime, nullable | Момент включения текущей паузы (NULL = не на паузе) |
| `sla_total_pause_seconds` | int | Накопленная пауза в **бизнес-секундах** (только рабочее время; см. ADR-017) |
| `sla_paused_by_status` | bool | Источник паузы: статус `WAITING_FOR_USER` / `ON_HOLD` |
| `sla_paused_by_reply` | bool | Источник паузы: последнее публичное сообщение от сотрудника |
| `created_at` | datetime | Дата создания |
| `updated_at` | datetime | Последнее обновление |

**Computed-поля в `TicketRead` (рассчитываются при сериализации, в БД не хранятся):**

| Поле | Тип | Описание |
|------|-----|----------|
| `sla_response_due_at` | datetime | Дедлайн первого ответа с учётом бизнес-часов и накопленной паузы |
| `sla_resolve_due_at` | datetime | Дедлайн решения, аналогично |
| `sla_response_remaining_seconds` | int | Сколько секунд до `response_due_at`. Может быть отрицательным = просрочено |
| `sla_resolve_remaining_seconds` | int | То же для resolve |
| `sla_remaining_seconds` | int | Остаток для **активной фазы**: response пока `first_response_at IS NULL`, иначе resolve |
| `sla_is_paused` | bool | Удобный alias `sla_paused_at IS NOT NULL` |

Расчёт — `backend/tickets/sla_service.compute_sla_state(ticket, now)`. Бизнес-часы — `backend/tickets/business_hours.py` (пн-пт 9-18 МСК).

### 3. FSM статусов (7 статусов, обновлено в v0.8)

```
НОВЫЙ ─────────────────────────────────┐
  │                                    │
  ▼                                    ▼
В РАБОТЕ ─────────────────────►  РЕШЁН (быстрое)
  │                                │         │
  ├──► ОЖИДАЕТ ОТВЕТА ──► В РАБОТЕ │         │
  │                                │         │
  ├──► ОТЛОЖЕНА ──────────► В РАБОТЕ         │
  │    (SLA на паузе)              │         │
  │                                │         │
  ├──► ВЫЕЗД ИНЖЕНЕРА ──► В РАБОТЕ │         │
  │    (SLA тикает)    └► ОЖИДАЕТ  │         │
  │                                │         │
  └──► РЕШЁН ◄─────────────────────┘         │
         │                                   │
         ├──► ЗАКРЫТ ◄───────────────────────┘
         │
         └──► В РАБОТЕ (переоткрытие)
```

**Автопереходы:**
- Первый ответ агента: Новый → В работе (автоматически)
- Ответ клиента: Ожидает ответа → В работе (автоматически)
- Все остальные переходы — вручную через выпадающий список статуса

### 4. Логика автоматизации

#### 4.1. Автоопределение классификаторов

При создании заявки система автоматически определяет:

```
title + description → ключевые слова →
  → product (если упоминается PASS24.Key → pass24_key, шлагбаум → pass24_auto, и т.д.)
  → category (по словарю: "SMS" → registration, "пропуск" → passes, ...)
  → priority (critical/high/normal/low)
  → sla_response_hours, sla_resolve_hours (из priority)
```

#### 4.2. Авто-SLA из приоритета

| Приоритет | sla_response_hours | sla_resolve_hours |
|-----------|-------------------|-------------------|
| critical | 1 | 4 |
| high | 2 | 8 |
| normal | 4 | 24 |
| low | 8 | 48 |

#### 4.3. SLA breach detection

```
sla_breached = (
    (first_response_at is None AND now > created_at + sla_response_hours)
    OR
    (resolved_at is None AND now > created_at + sla_resolve_hours)
)
```

### 5. Форма создания заявки (UX)

#### Шаг 1: Быстрый выбор (карточки)
Пользователь выбирает **что случилось** из визуальных карточек:
- 🚪 «Не могу попасть / проехать» → product=pass24_online, category=passes, priority=critical
- 📱 «Проблема с приложением» → product=pass24_online, category=app_issues
- 🔑 «Проблема с мобильным ключом» → product=pass24_key
- 🚗 «Шлагбаум не открывается» → product=pass24_auto, category=recognition
- 📋 «Проблема с пропуском» → product=pass24_online, category=passes
- ⚙️ «Проблема с оборудованием» → product=equipment, category=equipment_issues
- ❓ «Другой вопрос» → product=other

#### Шаг 2: Детали (форма)
После выбора карточки → форма с предзаполненными product + category:
- Тема (title) — обязательно
- Описание (description) — обязательно
- Название объекта (object_name) — рекомендуемо
- Телефон (contact_phone) — для экстренных
- Скриншот (screenshot) — опционально

#### Шаг 3: Подтверждение
Система показывает:
- Автоприоритет (можно изменить)
- Ожидаемое время ответа (SLA)
- «Отправить заявку»

### 6. Сравнение: текущая vs предлагаемая

| Аспект | Текущая | Предлагаемая |
|--------|---------|-------------|
| Классификация | 1 ось (category, 6 значений) | 5 осей (product, category, type, priority, source) |
| «Другое» | 82% заявок | <10% (за счёт product + refined categories) |
| Продукт | Не указан | 7 значений по реальному распределению |
| Тип обращения | Не указан | 5 типов (problem/question/request/incident/feature) |
| Источник | Не указан | 5 каналов (web/email/telegram/api/phone) |
| Объект | `object_id` (opaque) | object_name + object_address + access_point |
| Контакт | email через JWT | contact_name + email + phone + company |
| Техника | Нет | device_type + app_version + error_message |
| Ответственный | Нет | assignee_id |
| SLA | Хардкод 4/24ч | Автоматический из приоритета, breach detection |
| Автоприоритет | По ключевым словам | По словам + product + ticket_type |
